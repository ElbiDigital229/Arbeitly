# Arbeitly – Business Overview & Core Logic (Updated)

---

## 1. Core Concept

Arbeitly is a hybrid SaaS + service platform that helps candidates apply for jobs in Germany.

- **Free Plan** — Self-service toolkit (CV builder, application tracker)
- **Paid Plan** — Done-for-you execution (employee applies to jobs on behalf of candidate using AI-generated materials)

---

## 2. Value Proposition

### For Candidates:
- Build and export a professional CV
- Track all applications in one place
- Get done-for-you job applications (paid)
- Full transparency into everything happening on their behalf

### For Business (Arbeitly):
- Scalable service model via employee execution layer
- Structured, AI-assisted workflows
- Centralised performance tracking
- Revenue via paid plans + add-on quotas

---

## 3. Platform Architecture — 4 Layers

### 1. Candidate Layer
- Free users (self-service CV + tracker)
- Paid users (managed service with full execution)

### 2. Employee Layer
- Applies to jobs externally on behalf of candidates
- Uses AI-generated CVs and cover letters
- Manages assigned candidates' pipelines

### 3. Admin Layer
- Controls plans and quota limits
- Assigns candidates to employees
- Configures AI prompts, scraper, onboarding questions
- Manages transactions and platform settings

### 4. Intelligence Layer (AI + Automation)
- Job scraping per candidate via Apify (or equivalent)
- Candidate-job AI matching
- CV + cover letter generation and translation
- Language-aware CV routing (EN/DE)

---

## 4. Plan System

### 4.1 Free Plan
- CV builder (upload, edit, export) — export limit set by Super Admin
- Files (exported CVs only)
- Application tracker
- No AI enhancement
- No employee service
- Export limit triggers upgrade pressure

### 4.2 Paid Plans
- Everything in free, fully unlocked
- Application quota (e.g. 200 jobs) — set per plan by Super Admin
- Employee executes applications on behalf of candidate
- AI-generated CV + cover letter per job
- Language routing (EN/DE CV auto-selection)
- Job Discovery queue populated by per-candidate scraper
- FAQ interview preparation

---

## 5. Core Business Logic

### 5.1 Candidate ↔ Employee Mapping
- Each candidate is assigned to **one employee**
- Each employee manages **multiple candidates**
- Managed by Super Admin

### 5.2 Application Quota System
- Each paid plan has a fixed application quota (e.g. 200)
- Quota is tracked per candidate
- Quota is consumed **only when status = "Applied"** (employee has completed the application externally)
- Candidate self-added applications are **unlimited and never consume quota**
- When quota is exhausted:
  - Employee is blocked from adding new managed applications
  - Super Admin intervenes
  - Candidate can purchase add-on quota

### 5.3 Application Source Types

Every application is tagged with one of three sources:

| Source | Description |
|---|---|
| **Self** | Added by the candidate manually |
| **Employee** | Added and executed by the assigned employee |
| **System** | Generated by AI/scraper pipeline |

Each application must include: source tag, linked CV used, linked cover letter used, status timeline.

### 5.4 Employee Execution Model
The employee physically applies to jobs on behalf of the candidate. They use the candidate's **dummy application email + password** (created by the candidate during onboarding) to log in to external job portals and submit applications. After applying, they mark the status as "Applied" — this is when quota is consumed.

### 5.5 Application Visibility Rules

| User | Sees |
|---|---|
| **Candidate** | All applications — self-added, employee-added, system-generated |
| **Employee** | Only employee-added and system-generated (does NOT see candidate self-added) |
| **Super Admin** | All applications across all candidates |

### 5.6 CV Versioning System

Each CV evolves through stages with language variants:

- V1 → Original (uploaded by candidate or created from onboarding)
- V2 → AI Enhanced (global prompt)
- V3+ → Employee refined / job-specific variants
- Dynamic → Job-specific, language-specific variants

Every CV version carries a **language tag**: `EN` or `DE`.

### 5.7 Language Routing System

Admin defines global language routing rules. System detects job language requirements from the job description and applies rules automatically:

| Job Language State | Action |
|---|---|
| English only | Use English (EN) CV |
| German only | Use German (DE) CV — auto-generate via AI translation if none exists |
| English + German accepted | Auto-generate German (DE) version and use it |

Auto-generated German CVs are saved to the candidate's Files with a `DE` tag and linked to the application.

### 5.8 AI Prompt Control

Admin defines four global prompts that the system uses automatically:

1. **CV Enhancement Prompt** — used when employee enhances a CV
2. **Job Tailoring Prompt** — used for job-specific CV variants
3. **Cover Letter Prompt** — used for per-job cover letter generation
4. **Job Matching Prompt** — controls AI matching score per candidate

Prompt updates take effect immediately on all future actions. Prompt version history is maintained.

### 5.9 Per-Candidate Job Scraping

Jobs are scraped **per candidate**, not from a global pool.

- Super Admin configures scraper parameters per candidate: target roles, locations, industries, keywords, sources
- Admin sets the scraping cadence (daily, weekly, etc.)
- On activation, scraper runs automatically on the set schedule
- Results populate that candidate's Job Discovery queue in the employee portal
- Employee reviews matched jobs, selects appropriate CV/CL, applies externally, updates status

## 6. Data System Overview

### Candidate Data:
- Profile info (name, email, dummy application credentials)
- Marketing responses (free)
- Onboarding responses (paid)
- Plan details + application quota usage

### CV Data:
- All versions with source tracking (Candidate/Employee/System)
- Language tag per version (EN/DE)
- Job-specific variants

### Application Data:
- Job info, URL, description
- Status + timeline
- Source (Self/Employee/System)
- CV used (linked)
- Cover letter used (linked)

### Employee Data:
- Assigned candidates
- Application quota per candidate
- Activity logs

### Admin Data:
- Plans + application limits
- Transactions (Stripe + manual)
- AI prompts + version history
- Scraper configurations per candidate
- Onboarding question config

---

## 7. System Workflow (End-to-End)

### Free User Flow:
1. Sign up
2. Build/upload CV
3. Export CV (within limit)
4. Track applications manually
5. Hit limit → upgrade prompt

### Paid User Flow:
1. Sign up + payment
2. Create dummy application email during onboarding
3. Complete onboarding questionnaire
4. Assigned to employee by Super Admin
5. Admin configures and activates per-candidate scraper
6. Jobs scraped + matched appear in employee's Job Discovery
7. AI generates tailored CV + cover letter per job (with language routing)
8. Employee reviews, applies externally using candidate's credentials, marks "Applied"
9. Quota consumed per "Applied" application
10. Candidate tracks full progress in their portal

---

## 8. Summary

Arbeitly is a **job application operating system** where:

- AI generates optimised, language-appropriate application materials
- Employees execute applications at scale using those materials
- Admin controls intelligence via prompts, scraper config, and quota rules
- Candidates receive outcomes with full transparency
